# 클라이언트 사이드 암호화 (CSE) 조사 문서

 **목적**: 모바일 앱(Flutter/Dart)에서 민감 데이터를 클라이언트 사이드에서 암호화하여
 서버 운영자도 데이터에 접근할 수 없는 구조를 위한 방법론 조사

**방향**: 결정 전 단계의 조사 문서. 다향한 방법을 기술하며 결정은하지 않음

**핵심 조건**
 - 조건1: 서버에는 암호화 키가 절대 올라가지 않아야 함
 - 조건2: 기기 분실 시에도 복구 가능해야 함


---

## 1. 개요

### CSE란

데이터를 서버로 전송하기 전에 클라이언트(앱) 단에서 암호화하는 방식.
서버에는 암호화된 데이터(암호문)만 저장되며, 복호화에 필요한 키는 서버에 존재하지 않음.

```
일반 구조:
[평문 데이터] → [클라이언트에서 암호화] → [암호문만 서버로 전송/저장]
                        ↑
               [키는 클라이언트에만 존재 → 서버 운영자 접근 불가]
```

###  용어 설명

| 용어 | 설명 |
|---|---|
| **DEK** (Data Encryption Key) | 실제 데이터를 암호화하는 키 |
| **KEK** (Key Encryption Key) | DEK를 암호화하는 마스터 키(열쇠(DEK)를 또다른 열쇠로 잠구는방식) |
| **KDF** (Key Derivation Function) | 사용자가 입력한 비밀번호는 짧고 예측 가능해서 암호화 키로 바로 쓰기엔 취약 <br> 이 비밀번호를 가공해서 복잡한 암호화 키로 변환해주는 함수 |
| **FFI** (Foreign Function Interface) | Dart에서 직접 구현하기 어려운 기능이 필요할때  네이티브 라이브러리를 호출하는 인터페이스|

## 2. 키 관리 아키텍처

CSE에서 가장 중요한 설계 결정은 키를 어디에, 어떻게 보관하느냐이다.

### 2-1. Password-derived Key (비밀번호 파생 키)

사용자 비밀번호를 KDF에 통과시켜 암호화 키를 생성하는 방식.
서버에는 KDF에 필요한 `salt`와 KEK로 감싼 DEK(wrappedDEK)만 저장됨.
비밀번호 자체와 평문 키는 절대 서버로 전송되지 않음.

```
[비밀번호] + [salt] → [KDF] → [KEK]
                                    ↓
                      [KEK로 DEK를 암호화(wrap)]
                                    ↓
                      [wrappedDEK + salt → 서버 저장]
                      [DEK → 실제 데이터 암호화]
```

**서버 저장 정보**: 서버에는 salt + wrappedDEK만 저장. 비밀번호와 평문 키는 서버에 없음
**조건2**: 비밀번호만 알면 어느 기기에서든 복구 가능

**장점**
- 구현 난이도가 낮음
- 비밀번호만 알면 어느 기기에서든 복구 가능 (기기 독립적)
- Dart 단독 구현 가능

**단점**
- 보안이 비밀번호 강도에 완전히 종속, 약한 비밀번호 = 낮은 보안
- 비밀번호 분실 = 복구 불가 (서버도 비밀번호를 모름)
- 비밀번호 변경 시 DEK 재암호화 필요

**특징**
- KDF 연산 자체는 라이브러리가 제공하므로 직접 수학 구현 불필요.
- DEK/KEK 분리 구조 설계, wrappedDEK 저장/복호화 플로우, salt 관리 구현

---

### 2-2. Shamir's Secret Sharing (SSS, 비밀 분산)

키를 N개 조각으로 분산하고, K개 이상의 조각이 모여야만 원래 키를 복구할 수 있는 방식.

```
예: (2, 3) threshold — 3개 조각 중 2개만 있으면 복구 가능

[KEK] → SSS → [조각1: 서버] [조각2: 이메일] [조각3: 기기]

복구 시: 조각 2개 이상 조합 → [KEK 복구]
공격자: 조각 1개만 탈취 →  복구 불가
```

**서버 저장 정보**: 조각 하나만으로는 키 복구 불가. 서버가 조각 하나를 갖더라도 복호화 불가
**기기 분실 복구**: 조각이 분산 보관되므로 기기 분실 시 다른 조각으로 복구 가능

**장점**
- 조각 하나가 탈취돼도 키 복구 불가
- 조각을 이메일, 서버, 기기 등 다양한 곳에 자유롭게 분산 가능
- threshold 설정으로 보안과 복구 유연성 조절 가능

**단점**
- 조각 배포 채널 자체의 보안도 별도로 관리해야 함
- 일정개수 미달 시(조각 분실) 복구 불가
- 조각 무결성 검증 로직을 별도로 구현해야 함

**특징**
- 조각 배포/수집/검증 플로우 설계가 복잡함.


**참고자료**: [Wikipedia - Shamir's Secret Sharing](https://en.wikipedia.org/wiki/Shamir%27s_secret_sharing)

### 2-3. SRP (Secure Remote Password)

비밀번호 기반의 상호 인증 프로토콜
클라이언트와 서버가 수학적 교환을 통해 세션 키를 도출하며, 이 과정에서 비밀번호는 네트워크를 통해 절대 전송되지 않음.
서버에는 비밀번호 검증자(verifier)만 저장되며, 이것만으로는 비밀번호나 키를 복원할 수 없음.

```
[클라이언트]                              [서버]
비밀번호 보유                             verifier만 저장 (비밀번호 모름)
    ↓
A (공개값) ──────────────────────────→
           ←──────────────────────────  B (공개값) + salt
    ↓
수학적 연산으로 세션 키 S 독립 도출          수학적 연산으로 세션 키 S 독립 도출
    ↓                                        ↓
M1 (키 일치 증명) ──────────────────→  검증 후 M2 응답
← ─────────────────────────────────── M2 (서버 신원 증명)
    ↓
export_key → 데이터 암호화에 활용
```

**서버 저장 정보**: 비밀번호와 암호화 키가 서버로 전송되지 않음. 서버는 verifier만 보유
**기기 분실 복구**: 비밀번호만 알면 어느 기기에서든 동일한 key 도출 가능

**장점**
- 비밀번호가 네트워크를 통해 절대 전송되지 않음 (평문은 물론 해시값도 아님)
- 서버 DB가 탈취되어도 비밀번호 복원 불가 (verifier만 저장)
- 클라이언트와 서버가 서로의 신원을 상호 인증 (중간자 공격 방어)
- KDF를 클라이언트 사이드에서 실행 가능 → 서버 부담 감소
- Dart 전용 패키지 존재 (`dsrp`) → 단독 구현 가능

**단점**
- Password-derived Key보다 구현 복잡도가 높음
- SRP 자체는 인증 프로토콜이므로, 데이터 암호화를 위한 키 관리 구조(DEK/KEK)는 별도 설계 필요
- 비밀번호 분실 시 복구 불가 (서버도 비밀번호를 모름)

**특징**
SRP 프로토콜 자체는 `dsrp` 패키지가 처리하므로 직접 수학적 구현 불필요
단, 인증 후 도출된 `export_key`를 활용한 DEK/KEK 구조 별도 설계
백엔드에도 SRP 서버 측 구현 필요 verifier 저장, 라운드트립 처리




### 2-4. ZKP 기반 (Zero-Knowledge Proof, 영지식 증명)

키 소유 사실을 증명, 키 자체는 단 한 번도 공개하지 않는 방식.
인증 과정에서 서버가 어떤 비밀 정보도 수신하지 않음.

```
일반 인증: [사용자] → 비밀번호(또는 키) 전송 → [서버가 검증]
ZKP 인증:  [사용자] → "나는 이 키를 갖고 있다는 증명값" → [서버가 검증]
                         (키 내용은 서버가 알 수 없음)
```

**서버 저장 정보**:  키 자체가 서버에 전송되지 않음. 증명값만 전달
**기기 분실 복구**:  증명 기반이므로 기기와 무관하게 복구 가능 (비밀 보유 시)

**장점**
- 인증 과정에서 서버가 비밀 정보를 수신하지 않으므로 서버 탈취 시에도 키 유출 없음

**단점**
- 구현 난이도 어렵다고함
- 증명 생성 연산 비용이 매우 큼. 모바일 환경에서 응답 지연 발생 가능
- Dart 전용 라이브러리 없음 → FFI 바인딩 필요

**특징**
ZKP는 암호학 분야에서도 고급 주제에 속하며, 구현 실수가 보안 전체를 무너뜨릴 수 있는 위험이 높음.


**레퍼런스**: [Wikipedia - Zero-knowledge proof](https://en.wikipedia.org/wiki/Zero-knowledge_proof)

### 2-5. 니모닉(Mnemonic) 기반 키
 
사용자가 기억하거나 종이에 적어두는 12~24개의 단어 조합(니모닉)으로 키를 파생하는 방식.
암호화폐 지갑에서 표준으로 사용 중.
서버에는 어떤 키 관련 정보도 저장하지 않으며, 니모닉만 알면 어느 기기에서든 동일한 키를 복원할 수 있음.
 
```
[12~24개 단어] → [KDF] → [마스터 키]
                                      ↓
                              [DEK 파생 → 데이터 암호화]
서버: 암호문만 저장
```
 
**서버 저장 정보**: 암호문만 
**기기 분실 복구**: 니모닉만 있으면 어느 기기에서든 복구 가능
 
**장점**
- 서버에 키 관련 정보 일절 없음
- 니모닉만 보관하면 완전한 복구 가능 (기기 독립적)
- 암호화폐 지갑 생태계에서 검증
- 
**단점**
- 니모닉 분실 = 복구 불가 (서버에 없음)
- 니모닉을 디지털로 저장하면 보안 위협 (클라우드 메모, 캡처 등)
- UX가 불친절함 (일반 사용자에게 12개 단어 관리는 낯선 경험)

  
**특징**
Dart 전용 라이브러리 없음 → FFI 바인딩 또는 직접 구현 필요.
 
 
**레퍼런스**: [BIP-39 표준](https://github.com/bitcoin/bips/blob/master/bip-0039.mediawiki)
 
### 2-6. Device-bound Key + 니모닉 조합
 
Device-bound Key(평소 사용)와 니모닉(복구용)을 함께 운용하는 방식.
평소에는 하드웨어 보안 모듈이 키를 보호하고, 기기 분실 시에는 니모닉으로 복구.
 
```
평소:
[Secure Enclave 키] → 빠르고 안전한 암호화/복호화
 
분실 시:
[니모닉 12~24단어] → KDF → [동일한 키 복원]
 
서버: 암호문만 저장
```
 
**서버 저장 정보**: 암호문만
**기기 분실 복구**: 니모닉으로 복구 가능
 
**장점**
- 서버에 키 관련 정보 일절 없음
- 두 보안 레이어가 독립적으로 동작
- 
**단점**
- 구현 복잡도가 단독 방식보다 높음
- 니모닉 분실 시 복구 불가
- 두 가지 키 관리 플로우를 모두 설계해야 함
- 
**특징**
두 방식을 독립적으로 구현한 후 키 동기화 로직 추가 필요.
